# MigrationToExtension — План разработки

> **Цель:** Создать инструмент, который по трём входам (доработанная конфигурация, типовая конфигурация, реестр доработок) автоматически генерирует файлы расширения конфигурации с корректными аннотациями перехвата и diff-маркерами.

**Текущий статус:** Инфраструктура готова (CLAUDE.md, скиллы CFE, спецификации XML-форматов, session management). Нужно разработать кастомный пайплайн.

---

## Задача 1: Конфигурационный файл проекта

**Статус:** `[ ]`

**Зависимости:** нет

**Что делаем:**
Создать конфиг-файл (JSON или YAML), в котором задаются все входные параметры пайплайна. Агент и скрипты будут читать его для получения путей и настроек.

**Параметры конфига:**
- `customConfigPath` — путь к папке с выгрузкой доработанной конфигурации
- `baseConfigPath` — путь к папке с выгрузкой типовой (исходной) конфигурации
- `extensionOutputPath` — путь к папке, куда генерировать файлы расширения
- `registryPath` — путь к файлу реестра доработанных процедур
- `extensionName` — имя расширения (для NamePrefix и Configuration.xml)
- `extensionPurpose` — назначение: `Customization` / `Patch` / `AddOn`
- `compatibilityMode` — режим совместимости (напр. `Version8_3_24`)
- `workDir` — рабочая директория для промежуточных файлов (извлечённые процедуры)

**Файлы:**
- Создать: `config.json` (шаблон с комментариями)
- Создать: `config.example.json` (пример с реальными путями)

**Проверка:**
- [ ] Конфиг парсится без ошибок
- [ ] Все обязательные поля задокументированы

## Задача 3: Парсер BSL-модулей (извлечение процедур)

**Статус:** `[ ]`

**Зависимости:** нет

**Что делаем:**
Скрипт, который разбирает `.bsl` файл и извлекает каждую процедуру/функцию в отдельный файл. Это ключевой компонент — модули бывают до 40 000 строк, и передавать их целиком в контекст агента нецелесообразно.

**Логика парсинга BSL:**
- Процедура начинается: `Процедура ИмяПроцедуры(` или `Функция ИмяФункции(`
- Может иметь директивы контекста перед объявлением: `&НаСервере`, `&НаКлиенте`, `&НаСервереБезКонтекста` и т.д.
- Может иметь аннотации: `&Перед(...)`, `&После(...)`, `&Вместо(...)`, `&ИзменениеИКонтроль(...)`
- Заканчивается: `КонецПроцедуры` или `КонецФункции`
- Между процедурами может быть код инициализации модуля (область `#Область ИнициализацияМодуля`)
- Регионы/области: `#Область ИмяОбласти` ... `#КонецОбласти` — декоративные, не влияют на границы процедур
- Ключевые слова регистронезависимы в BSL (но по конвенции — CamelCase)
- Строковые литералы могут содержать `|` для многострочных строк
- Комментарии: `//` до конца строки

**Входные параметры:**
- `--input <путь к .bsl файлу>`
- `--output-dir <папка для извлечённых процедур>`
- `--procedures <список имён>` (опционально — извлечь только указанные, иначе все)

**Выход:**
Для каждой процедуры создаётся файл `<output-dir>/<ИмяПроцедуры>.bsl` содержащий:
- Директивы контекста (`&НаСервере` и т.д.) если есть
- Объявление процедуры/функции
- Тело
- `КонецПроцедуры`/`КонецФункции`

Также создаётся `<output-dir>/_index.json`:
```json
{
  "sourceFile": "CommonModules/УправлениеПечатью/Ext/Module.bsl",
  "procedures": [
    {
      "name": "СформироватьПечатнуюФорму",
      "type": "Процедура",
      "startLine": 145,
      "endLine": 230,
      "context": "&НаСервере",
      "isExport": true,
      "file": "СформироватьПечатнуюФорму.bsl"
    }
  ]
}
```

**Крайние случаи:**
- Процедура с `Экспорт` в объявлении
- Вложенные `Если`...`КонецЕсли` внутри процедуры (не путать с `КонецПроцедуры`)
- Препроцессорные инструкции `#Если Сервер`, `#Если Клиент`
- Многострочные параметры процедуры: `Процедура Имя(\n Параметр1, \n Параметр2)`
- Комментарии перед процедурой (считать частью процедуры если примыкают без пустой строки)

**Файлы:**
- Создать: `scripts/parse-bsl-module.py`

**Проверка:**
- [ ] Корректно парсит модуль с 10+ процедурами
- [ ] Правильно определяет границы вложенных конструкций
- [ ] Сохраняет директивы контекста
- [ ] Обрабатывает `Экспорт`
- [ ] Работает с файлами до 40 000 строк без проблем с памятью

## Задача 4: Оркестратор — извлечение пар процедур

**Статус:** `[ ]`

**Зависимости:** Задача 1, Задача 2, Задача 3

**Что делаем:**
Скрипт-оркестратор, который:
1. Читает конфиг (Задача 1)
2. Читает реестр доработок (Задача 2) → получает список модулей и процедур
3. Для каждого модуля из реестра:
   a. Находит `.bsl` файл в доработанной конфигурации
   b. Находит `.bsl` файл в типовой конфигурации
   c. Вызывает парсер BSL (Задача 3) для обоих файлов, извлекая только нужные процедуры
4. Формирует пары файлов для сравнения:
   ```
   workDir/
     УправлениеПечатью/
       base/                          # из типовой
         СформироватьПечатнуюФорму.bsl
         ПолучитьМакет.bsl
       custom/                        # из доработанной
         СформироватьПечатнуюФорму.bsl
         ПолучитьМакет.bsl
       _pairs.json                    # метаданные пар
   ```

**Формат _pairs.json:**
```json
{
  "modulePath": "CommonModules/УправлениеПечатью/Ext/Module.bsl",
  "objectType": "CommonModule",
  "objectName": "УправлениеПечатью",
  "pairs": [
    {
      "procedureName": "СформироватьПечатнуюФорму",
      "type": "Процедура",
      "context": "&НаСервере",
      "isExport": true,
      "baseFile": "base/СформироватьПечатнуюФорму.bsl",
      "customFile": "custom/СформироватьПечатнуюФорму.bsl",
      "status": "ready"
    }
  ]
}
```

**Обработка ошибок:**
- Процедура есть в реестре, но нет в типовой → новая процедура (не `&ИзменениеИКонтроль`, а собственная)
- Процедура есть в реестре, но нет в доработанной → предупреждение, пропуск
- Модуль не найден → ошибка, запись в лог

**Файлы:**
- Создать: `scripts/extract-pairs.py`

**Проверка:**
- [ ] Создаёт пары файлов для всех процедур из реестра
- [ ] Корректно обрабатывает отсутствующие процедуры/модули
- [ ] Формирует `_pairs.json` для каждого модуля

## Задача 5: Промпт для ИИ-агента — сравнение и генерация кода расширения

**Статус:** `[ ]`

**Зависимости:** нет (но нужны Задачи 3-4 для тестирования)

**Что делаем:**
Создать скилл (SKILL.md + промпт) для ИИ-агента, который получает пару процедур (исходная + доработанная) и генерирует код расширения с `&ИзменениеИКонтроль` и diff-маркерами.

**Вход агенту:**
- Исходная процедура (из типовой конфигурации)
- Доработанная процедура (из доработанной конфигурации)
- Метаданные: имя, тип (Процедура/Функция), контекст, Экспорт
- NamePrefix расширения

**Выход агента:**
BSL-код для модуля расширения:
```bsl
&НаСервере
&ИзменениеИКонтроль("СформироватьПечатнуюФорму")
Процедура Расш_СформироватьПечатнуюФорму(ПараметрыПечати, Коллекция) Экспорт
    // Полная копия тела доработанной процедуры
    // Блоки, удалённые из типовой, обёрнуты в #Удаление/#КонецУдаления
    // Блоки, добавленные в доработке, обёрнуты в #Вставка/#КонецВставки
    // Неизменённый код — без маркеров
КонецПроцедуры
```

**Правила генерации (описать в промпте):**
1. Декоратор `&ИзменениеИКонтроль("ОригинальноеИмя")`
2. Имя процедуры = `<NamePrefix><ОригинальноеИмя>`
3. Параметры и `Экспорт` — как в оригинале
4. Тело: полная копия из доработанной версии
5. Строки, которые были в типовой но удалены в доработке → `#Удаление` ... `#КонецУдаления`
6. Строки, которых не было в типовой но добавлены в доработке → `#Вставка` ... `#КонецВставки`
7. Строки одинаковые — без маркеров
8. Смежные удаления и вставки группировать (паттерн замены): `#Удаление`→`#КонецУдаления`→`#Вставка`→`#КонецВставки`
9. Отступы сохранять как в доработанной версии (табы)
10. Если процедура новая (нет в типовой) — без `&ИзменениеИКонтроль`, просто код

**Особый случай — выбор декоратора:**
- Процедура существует и в типовой, и в доработке, код различается → `&ИзменениеИКонтроль`
- Процедура существует только в доработке (новая) → без декоратора, собственная процедура расширения
- Процедура существует в обеих, код одинаков → пропустить (нет доработки)

**Файлы:**
- Создать: `.claude/skills/compare-procedures/SKILL.md`
- Создать: `.claude/skills/compare-procedures/prompt-template.md`

**Проверка:**
- [ ] Промпт понятен, содержит примеры всех случаев
- [ ] Результат агента — валидный BSL-код
- [ ] Маркеры расставлены корректно

## Задача 6: Сборщик модуля расширения

**Статус:** `[ ]`

**Зависимости:** Задача 5

**Что делаем:**
Скрипт, который собирает отдельные сгенерированные процедуры одного модуля обратно в единый `.bsl` файл расширения. Агент генерирует каждую процедуру отдельно — сборщик объединяет их.

**Логика:**
1. Читает все сгенерированные процедуры для одного модуля
2. Группирует по секциям/областям (если в исходном модуле были `#Область`)
3. Собирает в один `.bsl` файл с корректным порядком
4. Записывает в правильное место структуры расширения:
   - `CommonModules/<Имя>/Ext/Module.bsl`
   - `Catalogs/<Имя>/Ext/ObjectModule.bsl`
   - `Documents/<Имя>/Ext/ObjectModule.bsl`
   - `Documents/<Имя>/Forms/<Форма>/Ext/Form/Module.bsl`
   - и т.д.

**Файлы:**
- Создать: `scripts/assemble-extension-module.py`

**Проверка:**
- [ ] Собирает несколько процедур в один модуль
- [ ] Файл записывается по правильному пути в структуре расширения

## Задача 7: Главный оркестратор (end-to-end пайплайн)

**Статус:** `[ ]`

**Зависимости:** Задачи 1-6

**Что делаем:**
Главный скрипт, который запускает весь пайплайн от начала до конца:

1. Читает конфиг → валидирует пути
2. Вызывает `/cfe-init` → создаёт scaffold расширения
3. Читает реестр → получает список модулей и процедур
4. Для каждого модуля:
   a. Определяет тип объекта (Catalog, Document, CommonModule...)
   b. Вызывает `/cfe-borrow` → заимствует объект в расширение
   c. Извлекает пары процедур (Задача 4)
   d. Для каждой пары: вызывает ИИ-агента для сравнения (Задача 5)
   e. Собирает результаты в модуль расширения (Задача 6)
5. Вызывает `/cfe-validate` → проверяет результат
6. Генерирует отчёт: сколько модулей, процедур, ошибок

**Режимы работы:**
- `--dry-run` — только извлечь пары, без вызова агента
- `--module <имя>` — обработать только один модуль (для отладки)
- `--continue` — продолжить с места остановки (по статусам в `_pairs.json`)

**Файлы:**
- Создать: `scripts/migrate.py` (или `migrate.ps1`)

**Проверка:**
- [ ] End-to-end прогон на одном модуле
- [ ] Корректная структура расширения на выходе
- [ ] Отчёт содержит статистику
