# Migrate to Extension Skill

## Когда использовать

Когда пользователь просит перенести доработки конфигурации в расширение, выполнить миграцию доработок, или сгенерировать расширение из доработанной конфигурации.

## Предусловия

1. Файл `config.json` в корне проекта с полями:
   - `customConfigPath` — путь к выгрузке доработанной конфигурации
   - `baseConfigPath` — путь к выгрузке типовой конфигурации
   - `workDir` — рабочий каталог (по умолчанию `work`)
   - `extensionPrefix` — префикс имён процедур расширения
2. Файл `ОтчетОСравнении.txt` — подробный отчёт о сравнении из 1С (в корне проекта или путь уточнить у пользователя)

## Пайплайн

### Шаг 1: Сокращение отчёта о сравнении

```bash
python scripts/reduce-comparison-report.py "ОтчетОСравнении.txt" -o "РеестрИзменений.txt"
```

**Проверка:** файл `РеестрИзменений.txt` создан, содержит записи `## ТипОбъекта.ИмяОбъекта`.

### Шаг 2: Извлечение процедур

```bash
python scripts/extract-modified-procedures.py "РеестрИзменений.txt" -c config.json -o "РеестрПроцедур.txt"
```

**Проверка:** каталог `work/` создан, содержит файлы `*_typ.bsl`, `*_mod.bsl`, `*_new.bsl`.

### Шаг 3: Механическая генерация расширения (difflib)

```bash
python scripts/generate-extension-procedures.py work -c config.json --report "ОтчётГенерации.txt"
```

**Результат:**
- Файлы `*_ext.bsl` рядом с `*_typ.bsl` и `*_mod.bsl`
- JSON-отчёт `work/generation-report.json` с флагами `needs_review`
- Текстовый отчёт `ОтчётГенерации.txt`

**Проверка:** прочитать `ОтчётГенерации.txt`, убедиться что инвариант пройден для всех процедур.

### Шаг 4: ИИ-ревью крупных replace-блоков (субагенты)

Скрипт генерации (шаг 3) автоматически обрабатывает большинство случаев:
- **Рекурсивный sub-diff** — разбивает крупные replace-блоки, находя совпадения внутри
- **Comment-only детекция** — пропускает блоки, где код просто закомментирован

Процедуры с `skip_reason: "auto_refined"` или `"auto_comment_only"` уже обработаны.

**Для оставшихся** процедур с `"needs_review": true` — делегировать ревью субагентам:

1. **Прочитать** `work/generation-report.json`, отфильтровать процедуры с `needs_review: true`
2. **Для каждой процедуры запустить отдельный Agent** (субагент Sonnet):

```
Agent tool:
  model: "sonnet"
  prompt: содержит пути к 3 файлам + правила ревью (см. ниже)
```

3. **Запускать параллельно** — каждый субагент работает в своём контекстном окне, не потребляя основной контекст.

#### Промпт для субагента ИИ-ревью

Каждый субагент получает следующий промпт (подставить реальные пути):

```
Ты выполняешь ИИ-ревью процедуры расширения 1С.

Файлы:
- Типовая: {typ_file}
- Доработанная: {mod_file}
- Текущий результат: {ext_file}

Задача: прочитать все 3 файла. В _ext.bsl найти крупные replace-блоки
(пара #Вставка...#КонецВставки + #Удаление...#КонецУдаления).
Сравнить содержимое #Удаление и #Вставка построчно.
Если внутри большого блока реально изменены только несколько строк —
разбить на: неизменённый типовой код + точечные #Вставка/#Удаление.

Правила:
- НИКОГДА не изменять типовой код — копировать побайтово из _typ.bsl
- НИКОГДА не менять заголовок процедуры
- Маркеры #Вставка, #КонецВставки, #Удаление, #КонецУдаления — без отступов
- Порядок: сначала #Вставка (новый), потом #Удаление (старый)
- Если разбивка не даёт выигрыша — оставить как есть

Инвариант: убрать все #Вставка...#КонецВставки блоки с содержимым
и все #Удаление/#КонецУдаления аннотации (оставив содержимое).
Результат ДОЛЖЕН быть идентичен _typ.bsl. Проверь перед записью.

Записать результат обратно в {ext_file}.
Вернуть краткий отчёт: что изменено, сколько блоков разбито.
```

#### Почему субагенты

- Каждый субагент имеет собственное контекстное окно (~200K токенов)
- Основной контекст тратит только ~100 токенов на вызов + результат
- Субагенты могут работать параллельно (несколько Agent в одном сообщении)
- Масштабируется на любое количество процедур без риска переполнения

#### Предварительная оценка

Перед запуском субагентов — проверить секцию `estimation` в `generation-report.json`:
- `recommendation: "ok"` — запускать
- `recommendation: "warning"` — запускать с checkpoint
- `recommendation: "critical"` — предупредить пользователя, разбить на батчи

#### Протокол checkpoint (для больших объёмов)

Если `estimation.recommendation` = `"warning"` или `"critical"`, или процедур > 5:

1. **Создать** `work/checkpoint.json` перед началом ИИ-ревью:
```json
{
  "step": "ai_review",
  "started": "2026-03-08T14:30:00",
  "total": 12,
  "procedures": [
    {"name": "ИмяПроц", "module": "путь/к/модулю", "status": "pending",
     "typ_file": "...", "mod_file": "...", "ext_file": "..."}
  ]
}
```

2. **Перед запуском** каждого субагента — обновить статус на `"in_progress"`
3. **После завершения** субагента — обновить на `"completed"` (или `"failed"` с `"error"`)
4. **При рестарте сессии** — прочитать checkpoint:
   - `"completed"` → пропустить
   - `"in_progress"` → повторить (субагент мог не завершиться)
   - `"pending"` → запустить

Пример кода для оркестратора:
```python
# Чтение checkpoint
checkpoint_path = "work/checkpoint.json"
if os.path.exists(checkpoint_path):
    checkpoint = json.load(open(checkpoint_path))
    pending = [p for p in checkpoint["procedures"]
               if p["status"] in ("pending", "in_progress")]
else:
    # Создать из generation-report.json
    pending = [p for p in report["procedures"] if p["needs_review"]]
    checkpoint = {"step": "ai_review", "total": len(pending), "procedures": pending}
```

### Шаг 5: Сборка модулей

```bash
python scripts/assemble-extension-modules.py work -c config.json --report "ОтчётСборки.txt"
```

**Результат:**
- `_module.bsl` файлы в каждом каталоге модуля в `work/`
- Новые процедуры без префикса переименованы (ИмяПроц → Префикс_ИмяПроц)
- Вызовы переименованных процедур обновлены (ссылочная целостность)
- JSON-отчёт `work/assembly-report.json`

**Проверка:** прочитать `ОтчётСборки.txt`, убедиться что все модули собраны и переименования корректны.

### Шаг 6: Верификация

```bash
python scripts/verify-extension-modules.py work -c config.json --report "ОтчётВерификации.txt" --json
```

**Проверки:**
1. **Инвариант** — strip_markers(_ext) == _typ для всех процедур
2. **Префикс** — все процедуры в _module.bsl имеют префикс расширения
3. **Орфанные ссылки** — нет вызовов по старым (до переименования) именам
4. **Структура BSL** — парные Процедура/КонецПроцедуры, #Вставка/#КонецВставки

**Критерий:** exit code 0 = все проверки пройдены. При ошибках — показать детали и не продолжать.

### Шаг 7: Итоговый отчёт

После завершения всех шагов — вывести сводку:
- Сколько модулей обработано
- Сколько процедур: изменённых (_ext), новых (_new)
- Сколько прошли ИИ-ревью
- Сколько переименовано
- Результат верификации

## Обработка ошибок

- Если `ОтчетОСравнении.txt` не найден — спросить у пользователя путь
- Если `config.json` не найден — сообщить какие поля нужны
- Если скрипт завершился с ошибкой — показать stderr и предложить варианты
- Если инвариант не пройден — это критическая ошибка, показать diff и не продолжать
- Если checkpoint существует — предложить продолжить с места остановки

## Структура work/ после выполнения

```
work/
  generation-report.json            ← JSON-отчёт генерации (с estimation)
  assembly-report.json              ← JSON-отчёт сборки (с rename map)
  verify-report.json                ← JSON-отчёт верификации
  checkpoint.json                   ← прогресс ИИ-ревью (если использовался)
  ОбщийМодуль/
    ИмяМодуля/
      Модуль/
        ИмяПроц_typ.bsl            ← типовая версия
        ИмяПроц_mod.bsl            ← доработанная версия
        ИмяПроц_ext.bsl            ← процедура расширения с маркерами
        НоваяПроц_new.bsl          ← новая процедура (без пары)
        _module.bsl                ← собранный модуль (все _ext + _new)
  Документ/
    ИмяДокумента/
      Модуль_объекта/
        ...
      Forms/
        ИмяФормы/
          ...
```
